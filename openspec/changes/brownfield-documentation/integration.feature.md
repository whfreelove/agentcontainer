# Feature: Cross-capability lifecycle integration (integration)

## Integration Scenarios

`@integration:1` `@developer-initializes-project` `@developer-builds-container` `@developer-starts-container`
### Rule: Init-generated configuration files SHALL be consumable by build and up commands

#### Background

- Given the developer has run `agentcontainer init` in a project directory

`@integration:1.1`
#### Scenario: Build loads config files created by init

- Given init has generated `.agentcontainer/agentcontainer.conf` and `.agentcontainer/local.conf`
- When the developer runs `agentcontainer build`
- Then build SHALL use configuration values from both files for image construction

`@integration:1.2`
#### Scenario: Up parses devcontainer.json mounts generated by init

- Given init has generated `.devcontainer/devcontainer.json` with mount definitions
- When the developer runs `agentcontainer up` on Apple Container runtime
- Then up SHALL parse the mounts array from devcontainer.json into container CLI volume arguments

`@integration:1.3`
#### Scenario: Up extracts resource limits from init-generated devcontainer.json

- Given init has generated devcontainer.json with `runArgs` containing `--memory` and `--cpus` flags
- When the developer runs `agentcontainer up` on Apple Container runtime
- Then up SHALL extract resource limits from `runArgs` and pass them to the container runtime

`@integration:2` `@developer-builds-container` `@developer-starts-container`
### Rule: Build-produced images SHALL be available to up regardless of build runtime

`@integration:2.1`
#### Scenario: Native build image available for up

- Given the developer has run `agentcontainer build` on a runtime that matches the run runtime
- When the developer runs `agentcontainer up`
- Then up SHALL find the built image in the container registry and start a container from it

`@integration:2.2`
#### Scenario: Fallback-built image transferred and available for up

- Given the Apple Container builder daemon is unavailable
- And the developer has run `agentcontainer build` which built via Docker or Lima as fallback
- When build completes
- Then the built image SHALL be available in the Apple Container registry
- And up SHALL find the transferred image and start a container from it

`@integration:3` `@runtime-detects-platform` `@developer-builds-container` `@developer-starts-container` `@developer-opens-shell` `@developer-stops-container`
### Rule: Runtime detection result SHALL propagate consistently to all commands

`@integration:3.1`
#### Scenario: Detected runtime determines build docker-path

- Given the platform is detected as "darwin" with Apple Container available
- When the developer runs `agentcontainer build`
- Then build SHALL use the apple-container-shim as `--docker-path` for devcontainer CLI

`@integration:3.2`
#### Scenario Outline: Runtime override propagates to each lifecycle command

- Given the developer has set `CONTAINER_RUNTIME=docker` in local.conf
- When the developer runs `agentcontainer <command>`
- Then the command SHALL use `docker` as the container command regardless of auto-detection results

`@integration:4` `@developer-starts-container` `@developer-runs-agent` `@developer-opens-shell` `@developer-stops-container`
### Rule: Container naming conventions from up SHALL enable discovery by agent execution, shell, stop, and down

`@integration:4.1`
#### Scenario: Shell finds container created by up via name pattern

- Given up has created a container named `agentcontainer-${PROJECT_NAME}`
- When the developer runs `agentcontainer shell`
- Then shell SHALL locate the container by matching the `agentcontainer-` name pattern

`@integration:4.2`
#### Scenario: Stop finds devcontainer-created container via pattern fallback

- Given up has created a container via `devcontainer up` with name `vsc-${PROJECT_NAME}-<hash>`
- When the developer runs `agentcontainer stop`
- Then stop SHALL locate the container by matching the `vsc-` name pattern

`@integration:4.3`
#### Scenario: Down finds stopped container via label-based fallback

- Given a container was previously created and is now stopped
- And the container name does not match any known naming pattern
- When the developer runs `agentcontainer down`
- Then down SHALL locate the container by querying the `devcontainer.local_folder` label

`@integration:4.4`
#### Scenario: Agent execution finds container created by up via name pattern

- Given up has created a container named `agentcontainer-${PROJECT_NAME}`
- When the developer runs `agentcontainer` without a subcommand
- Then the default command SHALL locate the container by matching the `agentcontainer-` name pattern

`@integration:5` `@agent-auth-persists` `@developer-starts-container` `@developer-stops-container`
### Rule: Auth credentials in named volumes SHALL survive container removal and recreation

`@integration:5.1`
#### Scenario: Auth volume persists across down and up cycle

- Given a container is running with the `${PROJECT_NAME}-claude-home` named volume mounted at `~/.claude`
- And the agent has stored authentication credentials in the volume
- When the developer runs `agentcontainer down` followed by `agentcontainer up`
- Then the new container SHALL mount the same named volume with credentials intact

`@integration:5.2`
#### Scenario: Setup script symlinks workspace auth files into persistent volume

- Given a new container has started with the auth named volume mounted
- When `postCreateCommand` executes `setup.sh`
- Then setup.sh SHALL move `.claude.json` into the volume directory and create symlinks from the workspace to the volume

`@integration:6` `@developer-initializes-project` `@developer-runs-agent` `@developer-opens-shell`
### Rule: Init-configured agent and shell settings SHALL drive execution commands

`@integration:6.1`
#### Scenario: EXEC_AGENT placeholder substitution during agent execution

- Given init has set `EXEC_AGENT="nix-shell --run 'claude {}'"` in agentcontainer.conf
- When the developer runs `agentcontainer -- --help`
- Then the agent command SHALL replace `{}` with `--help` in the EXEC_AGENT template

`@integration:6.2`
#### Scenario: DEFAULT_SHELL from config determines shell command profile

- Given init has set `DEFAULT_SHELL=zsh` in agentcontainer.conf
- When the developer runs `agentcontainer shell` without `--shell`
- Then the shell command SHALL resolve the default profile to `zsh` and exec into the container with `/bin/zsh`

`@integration:6.3`
#### Scenario: remoteUser from devcontainer.json applies to both shell and agent exec

- Given init has generated devcontainer.json with `"remoteUser": "vscode"`
- When the developer runs `agentcontainer shell` or invokes the default agent command
- Then both commands SHALL extract `remoteUser` from devcontainer.json and pass it as the `--user` flag to container exec
